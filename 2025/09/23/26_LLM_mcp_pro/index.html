项目背景：为什么需要 MCP 工具？

大模型虽然擅长自然语言理解与生成，但在两类任务中存在明显短板：

  • 实时信息获取：如当前时间、实时天气等随时间变化的信息
  • 精确计算：如 BMI 计算、数学公式求解等需要精确结果的任务

MCP 工具（Model-Controlled Programs）正是为解决这些问题而生 —— 它作为大模型的 “外置能力扩展”，能让模型通过调用工具弥补自身短板。而 “自动调用” 则进一步解放了人力，让模型根据问题自主判断是否需要工具支持，无需用户手动触发指令。

本文介绍的方案基于 Ollama（本地大模型运行框架）实现，不仅支持显式指令调用（如/time查询时间），还支持模型自主决策调用（用户提问后，模型自动判断是否需要调用工具）。

项目地址：https://github.com/MiaoJieF/LLM_mcp_pro

一、第一步：创建 MCP 工具的核心流程

MCP 工具的本质是 “输入解析 + 业务逻辑 + 结构化输出” 的组合体。以项目中已实现的time（时间查询）和bmi（体质指数计算）工具为例，创建一个新工具需完成以下核心步骤。

1. 显式指令入口：支持用户手动触发

显式指令通常以/开头（类似命令行指令），便于用户快速调用。在代码层面，需实现一个指令解析函数，识别特定指令并执行对应逻辑。
以mcp_handler.py中的实现为例：

这段代码的核心逻辑是：

  • 识别以/开头的文本（判断为显式指令）
  • 对不同指令（/time、/bmi）编写专属处理逻辑
  • 返回标准化结果（MCPResult），包含 “是否处理成功” 和 “处理结果”

2. 程序化执行入口：支持模型自动调用

除了用户手动触发，工具还需支持模型通过程序自动调用。这就需要一个通用的执行函数，接收工具名称和参数，返回执行结果。
同样在mcp_handler.py中，execute_tool函数实现了这一功能：

与显式指令入口相比，程序化入口的特点是：

  • 工具名称更灵活（支持带/或不带，如time和/time都有效）
  • 参数通过字典传递，支持灵活的参数名（如height_m或height）
  • 强化参数校验（如身高单位必须为米，范围需合理）

3. 新增工具的 3 个关键步骤

基于上述框架，新增一个 MCP 工具（如查询天气的/weather）只需 3 步：

  • 在execute_tool中添加工具名称分支，实现核心逻辑（参数校验 + 业务计算）
  • 在handle_mcp_command中添加显式指令解析（可选，如需支持用户手动调用）
  • 文档化工具名称、参数格式和返回结果（方便模型理解如何调用）

二、第二步：实现大模型自动调用 MCP 工具的闭环

让模型 “自主决定是否调用工具” 是更高级的能力。核心思路是将决策权力交给模型，通过标准化的交互流程实现 “决策 - 执行 - 结果利用” 的闭环。

1. 设计思路：3 步完成自动调用

  • 模型决策：让模型判断是否需要调用工具，并用严格 JSON 格式输出决策（便于程序解析）
  • 工具执行：程序解析 JSON 决策，调用对应工具并获取结果，将结果以特殊标记（如 “（来自 MCP）”）写入对话历史
  • 生成答复：让模型基于包含工具结果的对话历史，生成最终的自然语言答复

3. 扩展建议：让 MCP 工具更强大

  • 丰富工具类型：新增数据库查询、API 调用（如天气、股票）、本地文件操作等工具
  • 结构化输出：为工具结果定义 JSON Schema，便于前端或其他系统二次利用
  • 优化决策稳定性：在模型决策阶段加入少量示例（few-shot），提升复杂场景下的决策准确性
  • 多工具协同：支持模型连续调用多个工具（如先查天气再计算出行时间）

四、总结

MCP 工具为大模型提供了 “扩展能力”，而自动调用机制则让这种能力的使用更加自然流畅。通过本文介绍的方法，我们可以快速创建新工具，并让大模型自主判断何时使用工具 —— 这不仅能弥补大模型的固有短板，还能大幅提升其在实际场景中的实用性。

该项目的完整代码已开源（参考项目结构），感兴趣的开发者可以直接基于此扩展，实现更复杂的工具调用能力。
